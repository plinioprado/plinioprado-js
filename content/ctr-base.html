More on the CTR Series prototypes

Rewriting 6 years old code led to some interesting notes:

React Hooks are fine. But using only functional components did make a difference in how clean the cod is. Classes were not missed as JavaScript is not a real OOP language.

Replacing Redux with React context/dispatch hooks is more a good alternative than an evolution. Redux is still a valid option, actually the final JavaScript code has a lot of similarities (not a surprise). Still, the dependency elimination was appreciated.

Adopting the latest version of most features was usually simple. Just a review of the documentation for the latest coding and dependencies. Routing was an exception. In addition to version compatibility issues, dropping Redux and using hooks required an adjustment on how to code it which took time. There is the benefit of vast and good documentation showing multiple ways to use it for your app, but it has a cost in time.

Bootstrap v5 no longer needing JQuery is great. But later it was fully replaced by react-bootstrap which works well.

There are visible simplifications in HTML/CSS coding due to no longer needing t support. But not much need to care about this when using create-react-app.

In the plain Javascript, the main change for the latest coding was async/await which already existed but was not widespread. It was good to use the newest syntax and it is better, but the previous version worked as well as it's the same Pormise under the hood.

Unlike the React client, the Node server just has to be correctly coded and provide the right responses to the right requests using the given SQL database. This is because React is a dominant client but Node is one of the good options for back-end.

The back-end focus was the implementation of a response composed of "data" and its display "format" because in an application the core components will require custom made coding, but there is always a swarm of auxiliary tables flying around them that shouldn't take much time to just define, add and move on.

Still in the "format", there are variations in the naming we usually see and remember for the properties used to code data validation. like "mandatory" vs. "required" vs. "notNull". Here they evolved to more strict following the HTML.
